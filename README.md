# База данных больницы — курсовая работа по базам данных

> Командный проект, университетский курс по базам данных  
> Задача: спроектировать реляционную базу данных для управления больницей — от ER-диаграммы до готовой схемы с данными

---

## О проекте

Курсовая работа по дисциплине «Базы данных». Мы разработали полноценную реляционную базу данных для больницы: спроектировали структуру с нуля, нарисовали ER-диаграмму в нотации Чена, реализовали схему на PostgreSQL и наполнили её тестовыми данными.

База данных охватывает основные сущности больницы — отделы, врачей, пациентов, приёмы, процедуры и диагнозы — и отражает реальные бизнес-правила: врач не может быть удалён без переназначения, один приём может включать несколько процедур, у каждого приёма может быть только один диагноз.

| PostgreSQL | СУБД, реализация схемы |
| SQL (DDL + DML) | создание таблиц, ограничений, вставка данных |

## Структура базы данных

База состоит из семи таблиц и одного составного типа.

**DEPARTMENT** — отделы больницы. Местоположение хранится как составной тип `Loc`, включающий номер корпуса и номер кабинета.

**DOCTOR** — врачи. Каждый врач привязан к одному отделу, имеет уникальные лицензию, телефон и email.

**PATIENT** — пациенты. Уникальный национальный ID, контактные данные, адрес.

**PROCEDURE** — медицинские процедуры с ценами. Название взято в кавычки, так как `PROCEDURE` — зарезервированное слово в PostgreSQL.

**APPOINTMENT** — приёмы. Связывает пациента и врача, фиксирует дату, время и статус (Completed / Scheduled / Cancelled / No-show).

**APPOINTMENT_PROCEDURE** — таблица-связка для отношения многие-ко-многим между приёмами и процедурами. Один приём может включать несколько процедур.

**DIAGNOSIS** — диагноз по итогам приёма. Связан с приёмом через `UNIQUE`-ограничение: у одного приёма может быть только один диагноз.

## Ключевые решения при проектировании

**Составной тип для местоположения.** Вместо двух отдельных полей `building_number` и `room_number` в таблице `DEPARTMENT` создали пользовательский тип `Loc`. Это позволяет работать с локацией как с одним объектом и подчёркивает, что эти два поля неразрывны по смыслу.

**Правила удаления через внешние ключи.** При проектировании думали о том, что должно происходить при удалении записей:
- Удаление пациента каскадно удаляет все его приёмы (`ON DELETE CASCADE`) — логично, без пациента его история не нужна
- Удалить отдел или врача напрямую нельзя, пока с ними связаны другие записи (`ON DELETE RESTRICT`) — защита от случайной потери данных
- Удаление приёма каскадно удаляет связанные процедуры и диагноз

**Связь многие-ко-многим через APPOINTMENT_PROCEDURE.** За один приём пациент может пройти несколько процедур. Реализовали это через промежуточную таблицу с составным первичным ключом из двух внешних ключей — стандартный подход для таких связей.

**Ограничения на уровне схемы.** Добавили `CHECK (Price >= 0)` для цены процедуры и `UNIQUE` на контактные данные врачей и пациентов — это исключает логически некорректные данные ещё до попытки вставки.

## ER-диаграмма

Диаграмма выполнена в нотации Чена (прямоугольники — сущности, ромбы — связи, эллипсы — атрибуты). Составной атрибут `Location` декомпозирован на `Building_number` и `Room_number`. Связи между сущностями:

- `WORKS_IN` — врач работает в отделе
- `INITIATES` — пациент инициирует приём
- `CONDUCTS` — врач проводит приём
- `INCLUDES` — приём включает процедуры
- `CONTAINS` — приём содержит диагноз

---

## Тестовые данные

В базу внесены:
- 10 отделов (кардиология, неврология, педиатрия и другие)
- 10 врачей с разными специализациями
- 10 пациентов
- 10 медицинских процедур с ценами от 150 до 5000
- 11 приёмов с разными статусами
- Связи процедур с приёмами и диагнозы

Данные подобраны так, чтобы покрыть разные сценарии: завершённые и отменённые приёмы, приёмы с несколькими процедурами, диагнозы разной тяжести.

---

## Выводы

Проект помог разобраться в полном цикле проектирования реляционной базы данных: от концептуальной модели (ER-диаграмма) до физической реализации (SQL-схема с ограничениями и данными).

Самое важное, что вынесла из этой работы — решения на уровне схемы имеют долгосрочные последствия. Неправильно настроенные правила удаления или отсутствие ограничений целостности сложно исправить после того, как база уже наполнена данными. Поэтому думать об этом нужно на этапе проектирования, а не после.
